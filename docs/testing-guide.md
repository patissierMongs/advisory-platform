# 완성단계 QA 테스트 가이드 (advisory-platform)

> "어느정도 완성단계라 직접 테스트에 들어가는 시점"에서 **UI/UX·버그·누락 패스**를 잡기 위한
> 가장 범용적이고 검증된 테스트 방법들 — 그리고 이 프로젝트(FastAPI + 빌드 없는 React SPA,
> 폐쇄망)에 **어디에·어떻게** 적용하는지를 정리한 실무 문서.

이 프로젝트의 현재 상태를 먼저 요약하면:

| 영역 | 현재 | 공백 |
|---|---|---|
| 백엔드 로직/API | **탄탄** — `pytest` 47개 + `smoke_test.py` 66체크. 서버 게이트·멱등성·매칭·피드 검증됨 | smoke_test가 pytest 밖(리포트 분산), 커버리지 미측정 |
| 프론트 SPA(관리자 4스텝 / 게시판) | **자동 테스트 0** — README도 "브라우저에서 수동 클릭 확인"이라 명시 | E2E·시각회귀·접근성 전무 |
| 자동화 | 로컬 수동 실행뿐 | **CI 없음**, 린트/타입체크 없음 |

→ 그래서 이 phase의 **최우선 투자처는 브라우저 E2E 자동화**다. 아래에서 방법론을 먼저 설명하고,
이 저장소에 대한 우선순위 실행안을 마지막에 붙인다.

## 0. 큰 그림 — 어디에 얼마나 투자할까 (테스트 피라미드/트로피)

모든 테스트를 똑같이 늘리면 느리고 깨지기 쉽다. 투자 비율의 감각:

```
        /\        E2E (소수)         ← 느림·비쌈, 진짜 사용자 동선만
       /  \       통합/API (다수)     ← FastAPI TestClient — 여기가 主전장
      /----\      단위 (많음)         ← 순수 로직(버전비교·정규식)
     /______\     정적분석(항상)      ← lint·type·format, 공짜에 가까움
```

- **핵심 원칙**: "빠르고 결정적인 것을 아래에 많이, 느리고 사람이 필요한 것을 위에 적게."
- 이 프로젝트는 이미 통합/단위가 두껍다. **비어 있는 꼭대기(E2E)와 바닥(정적분석)을 채우는 게** 가성비 최고.

## 1. 스모크 테스트 (Smoke Test) — 모든 QA의 첫 관문

**무엇**: "근본적으로 살아있나"를 1~2분 안에 전 구간으로 확인하는 얕고 넓은 테스트.
**왜**: 여기가 빨간불이면 다른 정밀 테스트는 시간 낭비다. QA 세션의 **가장 먼저** 돌린다.
**이 프로젝트에서**: 이미 `smoke_test.py`(66체크)가 업로드→추출→피드→매칭→발송→ack 전 구간을 검증.

```bash
python smoke_test.py     # 66체크 — 파이프라인 + 게이트
```

> 개선점: 커스텀 `check()` 대신 **pytest로 흡수**하면 리포트/CI가 하나로 통일된다(§9 실행안 P0).

## 2. E2E 테스트 (End-to-End, 브라우저 자동화) — 이 phase의 주력

**무엇**: **실제 브라우저**를 코드로 몰아 실제 사용자 동선을 그대로 재현(클릭·입력·이동·검증).
**왜 완성단계의 핵심인가**:
- 단위/통합이 다 통과해도 **버튼이 안 눌리고, 폼이 안 넘어가고, API 연동이 어긋나는** 건 브라우저에서만 드러난다.
- "누락 패스"(사용자가 밟는데 아무도 안 짜본 경로)를 **동선 단위**로 강제로 밟아 본다.
- README가 명시한 "React UI는 브라우저 클릭으로 최종 확인" — 이걸 **사람 대신 자동화**하는 게 정확히 E2E다.

**도구**: **Playwright**(권장). 이 환경엔 Chromium 사전 설치(`/opt/pw-browsers`), `.ds-build`에 이미 의존성 존재.
**이 프로젝트에서 자동화할 최우선 시나리오**:

1. **관리자 4스텝 해피패스** (`/admin`): PDF 업로드 → CVE 추출(비동기 폴링 대기) → 미등록 CVE로 게이트 닫힘 확인 → 피드 적용 → 게이트 열림 → 매칭 → 부서 발송 → 발송이력 반영.
2. **게시판 동선** (`/board`, 무인증): 관리자가 "게시판 게시" → 게시판 노출 확인 → 검색(자산번호·담당자·부서) → 댓글+조치상태 작성 → **발송 ack로 양방향 동기화** 확인.
3. **자산 임포트** (`/admin`): 지저분한 엑셀 업로드 → 컬럼 매핑 추천 → 커밋 → 자산·부서 자동생성 확인.

**핵심 기법** (E2E를 견고하게 만드는 규칙):
- **역할/텍스트 기반 셀렉터**(`getByRole`, `getByText`) — CSS 클래스 말고 사용자가 보는 것으로. UI 리팩터에 안 깨진다.
- **자동 대기(auto-wait)** 활용 — 임의 `sleep` 금지. 비동기 추출은 "상태 done 뜰 때까지" 조건 대기.
- **테스트 격리**: 매 테스트 깨끗한 시드 DB에서 서버 기동(`ADVISORY_SEED=true`, 임시 `ADVISORY_DATA_DIR`). 폐쇄망이라 외부망 0 — E2E도 완전 오프라인으로 돈다.

## 3. 탐색적 테스트 (Exploratory Testing) — UI/UX와 "모르는 문제"를 잡는 법

**무엇**: 대본 없이 **사람이** 목적을 갖고 자유롭게 파고드는 테스트. 단, 아무렇게나가 아니라 **차터(charter)** 로 구조화.
**왜**: UI/UX의 어색함·헷갈림·비일관성은 자동화가 못 잡는다. "이게 왜 여기 있지?", "이 상태에서 뭘 해야 할지 모르겠다" 같은 건 사람 눈에만 보인다.
**어떻게(세션 기반 탐색적 테스트, SBTT)**:
- 90분 이내 **세션**을 잡고, 한 문장 **차터**를 정한다. 예: *"게시판에서 담당자가 처음 들어와 자기 부서 권고문을 찾아 회신하기까지를 훑는다."*
- 발견은 즉시 **버그 로그**(재현 스텝·기대·실제·심각도)로 남긴다.
- 커버리지 휴리스틱으로 빠짐없이 훑기:
  - **SFDPOT**: Structure(구조)·Function(기능)·Data(데이터)·Platform(브라우저/해상도)·Operations(실제 사용시나리오)·Time(시간/순서).
  - **골디락스**: 값을 *너무 크게 / 너무 작게 / 딱 맞게* 넣어본다(업로드 크기, 빈 폼, 초대형 피드).
  - **중단/복귀**: 처리 도중 새로고침·뒤로가기·중복클릭·탭 이동.

**이 프로젝트 탐색 포인트**: 비동기 추출 중 UI가 뭘 보여주나 / 빈 상태(자산 0건, 매칭 0건)일 때 화면 / 에러 메시지가 "무엇을 하라"까지 알려주나 / 컬럼 매핑 추천이 지저분한 엑셀에서도 맞나 / 게시판 검색이 실제로 걸러지나.

## 4. 경계값·부정·엣지 테스트 — "누락 패스"의 정면 대응

**무엇**: 정상값이 아니라 **경계·비정상·실패** 입력을 의도적으로 넣는다.
**왜**: 개발자는 해피패스를 짠다. 버그와 누락 패스는 거의 항상 **실패 쪽 쌍둥이 경로**에 숨어 있다.
**황금 규칙 — "모든 해피패스에는 실패 쌍둥이가 있다"**. 각 동선마다 아래를 짝지어 확인:

| 해피패스 | 반드시 같이 볼 실패/경계 |
|---|---|
| PDF 업로드 성공 | 같은 파일 재업로드(중복 SHA256 → 409), 30MB 초과, PDF 아닌 파일, 빈 파일 |
| CVE 추출 | 추출 버튼 **연타**(중복 디스패치 차단), 텍스트 안 뽑히는 스캔 PDF, 서버 재시작 중 stuck 복구 |
| 매칭 진행 | **미등록 CVE 남은 채 매칭**(409 게이트), 자산 0건, 버전 애매(candidate) |
| 발송 | **같은 조합 재발송**(멱등 — 중복 발송 0), SMTP 미설정/실패(→FAILED, SENT 아님) |
| 게시판 회신 | 빈 부서/이름, 대량 댓글, 존재하지 않는 권고문 id 직접 접근 |

**기법**: 등가분할(Equivalence Partitioning)·경계값분석(BVA)·에러추정. **부정 테스트(negative test)** 는 "막혀야 할 게 제대로 막히는지"를 본다 — 이 시스템의 두 서버 게이트가 대표적.

## 5. 상태전이 테스트 (State-Transition / 모델 기반) — 누락 패스를 체계적으로

**무엇**: 시스템의 상태기계를 그리고 **모든 전이(합법/불법)** 를 표로 만들어 하나씩 친다.
**왜**: "누락 패스"를 감이 아니라 **완전열거**로 찾는다.
**이 프로젝트**: `AdvisoryStatus` = UPLOADED → EXTRACTING → EXTRACTED → NEEDS_CVE_UPDATE → MATCHED → NOTIFYING → COMPLETED (+ARCHIVED). 각 전이에 대해:
- **정상 전이**가 UI/서버 양쪽에서 일어나나?
- **불법 전이**(예: NEEDS_CVE_UPDATE에서 바로 발송)가 막히나?
- 각 상태에서 화면이 **다음에 뭘 해야 하는지** 안내하나? (빈 상태·막힌 상태의 UX)

`ack_status`(NONE/IN_PROGRESS/DONE/UNABLE)와 `NotificationStatus`(PENDING/SENT/FAILED/ACKED)도 같은 방식으로.

## 6. 회귀 테스트 + CI 자동화 — 고친 걸 다시 안 깨지게

**무엇**: 버그를 고칠 때마다 **그 버그를 재현하는 테스트를 먼저** 추가하고(고치면 통과), 전 스위트를 매 변경마다 자동 실행.
**왜**: 완성단계 QA는 "고치다가 다른 걸 깨는" 두더지잡기가 된다. 회귀 스위트 + CI가 그물이다.
**어떻게**: 발견→(재현 테스트 추가)→수정→CI가 앞으로 영구 감시. **GitHub Actions**로 push/PR마다 `pytest` + `smoke` + `E2E` + lint 자동 실행(§9 P1).

## 7. 시각적 회귀 + 접근성 — UI/UX 품질의 자동 측정

- **시각적 회귀(Visual Regression)**: Playwright `toHaveScreenshot()` 로 기준 스냅샷을 잡아두고, 이후 **레이아웃·CSS 깨짐**을 픽셀 비교로 감지. 기능 테스트가 못 보는 "화면이 어긋났다"를 잡는다. `.ds-build/verify.mjs`가 이미 컴포넌트 스크린샷을 찍고 있어 확장 용이.
- **접근성(a11y)**: `@axe-core/playwright`로 대비·라벨·역할 위반 자동 검출 + **키보드만으로 전 동선** 통과(포커스 순서·엔터 제출). 공공기관 대상 시스템이면 특히 가치.

## 8. 바닥 다지기 — 정적분석·커버리지·속성기반 (거의 공짜인 버그 방지)

- **정적분석/린트/포맷**: `ruff`(Python lint+format), `mypy`(타입), `.ds-build`엔 `tsc`(strict 이미 설정). 실행 전에 **버그 클래스 통째로** 걸러낸다. 폐쇄망 런타임과 무관한 개발/CI 전용이라 안전.
- **커버리지로 누락 패스 찾기**: `pytest --cov=app` 로 **안 밟힌 분기**를 리포트 → 그게 곧 코드상 "누락 패스". 숫자 자체보다 **빨간 줄 목록**을 QA 대상으로 쓴다.
- **속성기반 테스트(Property-based, `Hypothesis`)**: 순수 로직에 무작위 입력을 쏟아 **사람이 못 떠올린 엣지**를 찾는다. 이 프로젝트의 최적 후보 2곳:
  - `core/versioning.py` — 버전 비교기(23H2/2021/124/DC 2022): "a<b, b<c면 a<c" 같은 불변식.
  - `core/extract.py` — CVE 정규식: 어떤 변형(공백·구분자)이든 표준형 왕복(round-trip)이 유지되는지.

## 9. 이 저장소에 대한 우선순위 실행안 (요청 시 구축)

방법론을 이 프로젝트에 실제로 심는 순서. **위에서부터 가성비 순.**

| 순위 | 항목 | 무엇을 추가 | 왜 |
|---|---|---|---|
| **P0** | **Playwright E2E** | `tests/e2e/`에 `playwright.config.ts`(webServer로 `uvicorn` 시드 기동) + §2의 3개 시나리오. Chromium 사전설치 재사용 | 가장 큰 공백. README가 요구한 수동 확인을 자동화 |
| **P0** | **smoke → pytest 흡수** | `smoke_test.py`를 `tests/test_smoke_e2e.py`로(또는 얇은 래퍼) | 리포트/CI 단일화 |
| **P1** | **CI (GitHub Actions)** | `.github/workflows/ci.yml`: `pytest`+`smoke`+`playwright`+`ruff`+`mypy`+`tsc` | 회귀 그물. 현재 CI 0 |
| **P1** | **정적분석** | `pyproject.toml`에 `ruff`·`mypy` 설정, dev 의존성 추가 | 값싼 버그 방지 |
| **P2** | **시각회귀 + a11y** | Playwright `toHaveScreenshot` + `@axe-core/playwright` | UI/UX 자동 측정 |
| **P2** | **커버리지 + 속성기반** | `pytest-cov`, `versioning`/`extract`에 `Hypothesis` | 누락 패스·엣지 발굴 |

**주의(폐쇄망)**: 위 테스트 도구는 전부 **개발/CI 전용**이며 운영 오프라인 번들(`vendor/wheels`)과 분리된다.
런타임 실행 경로(§start.sh)는 그대로 외부망 0을 유지한다.

## 10. 실전 QA 세션 순서 (완성단계에 그대로 따라 하기)

1. **그린 베이스라인**: `pytest` + `smoke_test.py` 통과 확인. 빨간불이면 여기부터.
2. **E2E 해피패스**(§2) 자동 실행 — 핵심 동선이 끝까지 도나.
3. **탐색적 세션**(§3) 화면별 1개 차터씩 — 버그 로그 작성.
4. **엣지/부정 패스**(§4 표) 체크리스트로 훑기 — 실패 쌍둥이 전부.
5. **시각회귀 + 키보드/a11y**(§7) 스윕.
6. 발견 → **재현 테스트 추가** → 수정 → **CI가 영구 감시**(§6).

### 버그 로그 템플릿
```
[제목] 한 줄 증상
심각도: Blocker / Critical / Major / Minor / Trivial
재현: 1) ... 2) ... 3) ...
기대: ...
실제: ...
환경: 브라우저/해상도/시드 여부
```

## 부록 A. 명령어 빠른 참조

```bash
# 백엔드 단위/통합
pytest tests

# 전 구간 스모크(66체크)
python smoke_test.py

# (P0 구축 후) 브라우저 E2E
#   서버는 config의 webServer가 ADVISORY_SEED=true로 자동 기동
npx playwright test               # tests/e2e/

# (P1 구축 후) 정적분석
ruff check app tests && mypy app
```
